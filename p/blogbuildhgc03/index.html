博客框架Hugo

框架软件下载

👉 Hugo官网 https://gohugo.io

进入官网后，我们需要进入其GitHub页面进行下载，如果您从未登陆使用过GitHub这个网站，可以参考下列图片步骤：

您可能需要知晓的是，版本号并非越新越好，有时候版本不同可能会造成后续多个软件之间的冲突，以及更新的版本可能会有一些技术细节上的变动带来后续调试上的变动，本文以v0.131.0为例，适配系统环境的文件为【hugo_extended_v0.131.0_windows_amd64.zip】。

建议养成良好的文件管理习惯，将Hugo解压缩到专门的文件夹，建议全英文，无特殊符号。

主题安装

到这里我们已经可以使用Hugo了，接下来，我们要使用命令让Hugo创建出一个博客文件夹并生成基本的博客框架数据。在Hugo所在的文件夹地址下使用cmd调出命令行（您也可以选择在搜索中打开命令行再使用cd命令进入到Hugo所在的文件夹地址），在命令行输入hugo new site myblogname，注意，myblogname将作为博客数据文件夹名称，您可以将其改为其它的名称，例如hugo new site aaa、hugo new site bbb，名称全英文，无特殊符号。

与此同时，您可以看到多了一个叫做myblogname（或您给文件夹取的其它名字）的文件夹，且其中已经有很多数据——以及空文件夹。这就是我们博客的原型，毛胚房形态，具备核心的博客文件夹并且有基础配置，但除此之外空空如也，因此我们需要对此装修，也即主题。

这可以说是Hugo的核心功能之一，它给出了一个目录文件模板，而那些主题们需要遵循Hugo给的规则来配置主题内参数，而作为使用者，我们可以相对方便、低成本的切换各个主题，并按照其规则进行主题装修。

我们回到👉 Hugo官网 https://gohugo.io ，浏览并下载您喜欢的主题，在这里Stack主题v3.26.0为例。需要注意的是，不同的主题后续进行配置时可能会有些细节上的不同，如果您是第一次鼓捣安装博客，且希望花费较少的试错成本，您可以和本文一样，也使用Stack主题练手，等您能相对熟练进行相关配置时，再选择其它主题。

基本参数调试

在这里，我们可以选择配置全局环境变量（简单来说即任何文件目录下我们都能使用hugo.exe软件），进入到高级系统设置—>环境变量—>在全局系统变量的Path变量中增添hugo.exe所在的路径；

也可以选择一些偷懒的办法，将hugo.exe软件复制一份到对应的博客文件夹地址下，但并不推荐这种做法，hugo的启动软件hugo.exe本体有近100m大小，当后续我们需要管理的博客变多，或是涉及到自动部署时，可能需要进行一些额外的操作，依旧建议您配置全局环境变量。

在大多数主题中，作者都会给出exampleSite文件夹，即基本上已经配置好的样例，我们将核心配置config.yaml（后缀名也可能是.toml）——本文的主配置文件名为hugo.yaml，和content文件夹放到我们自己的博客文件夹主目录下，最基本的主题结构到这里就完成了。

需要说明的是，Hugo的机制为：渲染博客主题时，会优先检查我们博客主文件夹下进行的配置，如果我们并未进行设置，则会采用themes下的配置，因此建议在后续修改对应文件时，将themes下模板文件复制到我们自己的博客主目录文件夹下进行修改。

对核心配置hugo.yaml进行检查和调整，我们在调整的过程中可以随时通过hugo server -D命令进入对应网址实时预览调整后的结果。以下是一些基本属性，更详细的设置可以参考您所下载的主题的作者所提供的参考文档，如果您使用的是Stack主题，它的参考文档地址是 👉 Stack主题参考文档 https://stack.cai.im/config/menu 。

项目部署

前言

经过上述对Hugo框架的折腾，我们获得了一个本地的博客页面，接下来，我们需要将其部署到服务器上，并获取域名，实现联网效果，来让其它用户也能在线访问到我们的博客，本文采用GitHub进行一系列部署。

Git与GitHubDesktop

我们需要Git这个中介将我们的博客（本地版）上传到GitHub。

👉 Git官网 https://git-scm.com

事实上，如果您是Windows或Mac系统，我会更推荐您下载GitHub Desktop，GitHub Desktop是图形用户界面，来简化使用 Git 和 GitHub 的过程，即您不需要直接面对全是命令行的界面，而是像我们如今所用的简单软件那样，通过图形按钮等来实现我们的需求。

👉 GitHub Desktop官网 https://desktop.github.com/download/

工具准备齐全，我们下列正式进入到GitHub的使用。

GitHub

👉 GitHub官网 https://github.com

首先我们需要一个账号，建议用一个新账号来管理博客，推荐使用邮箱注册，不想与现实身份强联系的可以考虑海外邮箱，试手拿着玩的可以用临时邮箱。

这里需要注意账号名的选取，采用该方案创建的博客，如果您对折腾域名的意愿不强，那么我们获取的域名将与你的账号名强联系，例如本博客GitHub账号名为aaa，那么最后域名会是https://aaa.github.io形式。

账号注册成功后，我们注册一个新公开仓库repository，后续放博客的public文件夹，实际上整个public文件夹就是Hugo创建出来的静态页面。

考虑到安全性，我们再建立一个私有仓库repository，只有我们自己能看见，用于放置博客的数据源文件（您可以理解为除了Pulic文件夹以外的其它文件夹），仓库的创建和上述基本相同，在Choose visibility选项选择Private即可，这里对于仓库名没有过多要求，以见名知意为佳，例如您的博客名为aaa，可以给该仓库取名为aaaHugoBlogSource。

到这里，我们获取了域名，现在可以对应修改我们博客配置文件hugo.yaml中的访问地址了，修改baseurl属性：

baseurl: https://您的用户名.github.io/

自动化生成Public文件

在常规的操作中，我们需要使用hugo命令生成public静态页面文件夹，再将该public文件夹上传到GitHub的公开仓库。为简化这一过程，我们期望的是只需要管理文章内容以及一些博客装修后修正的部分，即仅管理源代码，而减少掉手动生成public静态页面文件夹的工作。

因此我们将采取GitHub Action并使用一段自动代码，实现仅管理私有仓库，而动态生成public静态页面文件夹，并且能够自动的上传更新到对应的公开仓库，从而实现页面更新。

我们的自动化依赖一段代码完成，而第一步是要给权限，书面说法为“GitHub 提供一个令牌Token，可用于代表 GitHub Actions 进行身份验证”。

设置Token具体操作：打开GitHub，点开个人界面的Settings，找到Developer Settings，有一个钥匙图标Personal access tokens (classic)，点进其中的Tokens (classic)，点击Generate new token (classic)，备注Note是给自己看的注释，到期时间Expiration选择No expiration（如设置有效期过了有效期需要重新给权限），并勾选repo和workflow这两部分。

请注意，生成的那串Token只会显示一次，所以您可以单独创建一个文本文档进行保存记录，否则就只能重新设置。

因为这段Token会被我们写进接下来的自动化代码文件里，所以我们要在这里再进行一个操作，在我们放源代码的私有仓库设置一个Secrets：

具体操作：点开我们刚刚创建的私有仓库，点开仓库的Settings，找到Security下的Secrets and variables，点进其中Actions，创建一个New repository secret，Secret那个框里粘贴进去刚刚你复制的那串Token，这里的Name建议符合全英文+简洁原则，例如小编这里写的是TOKEN。

下面来到本部分真正的核心——自动化代码部分，我们回到电脑本地文件夹，存放博客所有文件的位置，新建一个文件夹”.github“，并在”.github“内创建一个子文件夹”workflows“，我们的自动化代码就放在”workflows“文件夹下，创建一个空白文档名为deploy.yml，粘贴以下代码：

  deploy:
    runs-on: ubuntu-24.04 # 之前用这段代码一直有问题，网友说可能是因为这个ubuntu的版本问题，于是改成了这个版本

    steps:
      - name: Checkout repository
        uses: actions/checkout@v2
        with:
          submodules: true
          fetch-depth: 0

      - name: Set up Hugo
        uses: peaceiris/actions-hugo@v2
        with:
          hugo-version: "latest"

      - name: Build and Deploy
        run: |
          hugo -D

      - name: Deploy to GitHub Pages
        uses: peaceiris/actions-gh-pages@v3
        with:
          PERSONAL_TOKEN: ${{ secrets.TOKEN }} # 你的个人访问令牌，按照变量名来修改，上述用的是TOKEN就写TOKEN，如果在创建secrets名字叫做ccc，那么这里就写secrets.ccc

          PUBLISH_BRANCH: main # 或者是你的GitHub Pages分支
          PUBLISH_DIR: ./public
          commit_message: ${{ github.event.head_commit.message }}

使用GitHubDesktop上传数据

数据准备好了，仓库准备好了，并且到这里为止我们后续只需要管理私有仓库而不再用关心Hugo创建Public静态页面文件夹的工作，接下来我们只需要将大象放进冰箱里就好了。

登录GitHub Desktop，如果您已经登录了GitHub账号会提示您进行关联登录，因为GitHub代码仓库管理的逻辑其实是：【线上仓库有文件——克隆（下载）文件到本地——本地文件有更新——确认提交更新后的代码】，因此我们需要准备一个文件夹先让GitHub Desktop把线上仓库克隆下来到本地，再更新这个文件夹，用GitHub Desktop提交更新，建议您将这个文件夹和我们先前创建的博客文件夹放在同一级目录，方便后续复制粘贴，您也可以选择将整个源文件都更新到由GitHub Desktop管理的文件夹，但更建议您留份原件。

在正式上传前，我们先设置不需要上传到私有仓库中的文件，例如public文件、Hugo.exe文件，我们可以创建”.gitignore“文件，在上传的时候GitHub Desktop将自动忽略我们这个文件里写的内容，”.gitignore“文件位置写在博客文件夹根目录下。

进行文件夹的内容更新后，打开GitHub Desktop会看到”Changes“那一块多出来很多东西，是GitHub Desktop对比比较出文件更新后的内容，我们在”Summary“部分可以写好本次改动的原因，commit to main，然后确认提交push origin，如果失败了大概率是网不行，建议换个节点。

稍等片刻，代码生效需要一些时间，可以点开GitHub仓库的Actions查看部署deploy情况。

到这一步应该就能完成整个的自动化操作了，如果没有，请看下一步。

公开仓库检查

如果上传出现了问题，可以检查一下GitHub中管理Public文件夹的公开仓库，在这里部署不成功可能是有两类问题：

  • 你的公开仓库其实还不算建立成功
  • 公开仓库里文件多了

第一种情况，是创建仓库后需要有一个上传的操作，可以任意写入一点内容，这个时候仓库才是真正的创建成功，否则在变更私有仓库数据时，无法正确部署到公开仓库——因为它找不到。注意写入之后要对公开仓库内容进行清空，否则也可能出现博客页面出错等问题。

第二种情况，大多发生在自行调试公开仓库过程中手动对公开仓库添加了数据，按照我们的流程，公开仓库中的所有数据都应该是由私有仓库通过自动部署上传到公开仓库的，不应该存在我们手动的部分，这个时候可以检查仓库内容，如果发现有除了跟私有仓库有联系的代码——Code页面每一行提交的文件中间有亮着蓝色的部分，例如小编这边显示是”用户名/xxxxxxx(私有仓库名)@xxxxxxxx(一串数字)“，还出现了部分没有联系的，就属于该种情况，如果您出现了这样的情况，恭喜你，现在打开GitHub Desktop，按照克隆操作把公开仓库拉到本地，然后把文件清空再上传，即把公开仓库清空。

确认公开仓库没有问题（没有多余文件）后，使用GitHub Desktop对私有仓库数据进行更新，因为自动化代码触发貌似需要代码有变动，所以这里可以任意新增一个测试博文文件（\content\post部分添加新的文章），再再再更新上传，来触发代码生效。

此时应该就能正常打开博客页面了。

Cloudflare

选择GitHubPage搭建是很方便，不过github本身的网络访问问题可以进一步优化，这里涉及到简单的CDN(内容分发网络，Content Delivery Network)概念，简单来说CDN提供商们可以实现托管我们的博客地址，用它们自家的节点进行分发操作，优化用户的访问体验，并且允许给我们的博客添加域名——即多个域名均指向我们的博客，也可以实现隐藏我们服务器的真实地址的效果。而 CloudFlare 就是一家CDN提供商。

进入官网注册账号后，在左侧导航栏找到【计算 Compute】分类下的【Workers & Pages】，创建新的应用程序并选择导入现有 Git 存储库，将我们在GitHub放置博客页面的公共仓库进行授权，一步步点击最后部署等待完成。

这时CloudFlare会给我们一个.works.dev的域名（或是.pages.dev），和github给我们分配的域名等效。

到这里最基础的优化就完成了，Cloudflare的功能当然远不止此，后续我们还可以购买域名，再在CloudFlare上进行绑定，在本篇内容中暂时不进行讨论。

炸掉重来

如果您在数据上传的部分卡住，或许是折腾GitHub的仓库时里面上传的数据出了些问题，如果怎么折腾都还是不对，您可以选择将仓库删除再重新创建。删除的步骤：点进对应仓库Repositories，仓库Settings，划到最下方Danger Zone，Delete this repository。

在这个过程中，建议您始终备份您的本地博客文件夹——包括文章内容和装修数据。
